Three Architecture Approaches

Stateless Python API + local-first browser storage — pick your frontend stack and coupling style

A

SvelteKit + FastAPI ★ Recommended

Clean reactive stores map directly to the notebook widget pattern. Lightest runtime, best suited for data-heavy reactive UIs.

Browser (SvelteKit)
Svelte stores ↔ Dexie.js (IndexedDB)
Plotly.js / ECharts charts
ZIP export/import
Python (FastAPI)
POST /preprocess → returns events+metrics JSON
POST /validate-schema → schema lint
bodaqs_analysis unchanged as library

Pros

  • Smallest runtime bundle (~5kb base)
  • Reactive stores ≈ existing widget contracts
  • First-class TypeScript + great tooling
  • SvelteKit routing suits multi-page app

Cons

  • Smaller ecosystem than Vue/React
  • Less off-the-shelf UI component libraries
B

Vue 3 + FastAPI

Composition API + Pinia stores. Rich component ecosystem including data grids, tabs, and form controls that mirror the existing ipywidgets UI closely.

Browser (Vue 3 + Pinia)
Pinia stores ↔ Dexie.js (IndexedDB)
ECharts (vue-echarts) charts
ZIP export/import
Python (FastAPI)
Same stateless API surface as A

Pros

  • Largest component library ecosystem
  • PrimeVue / Naive UI for data grids/tables
  • Strong corporate backing (Evan You)
  • Gradual learning curve

Cons

  • Heavier runtime than Svelte
  • More boilerplate for reactive data flows
C

Preact + FastAPI

Minimal React-compatible layer. Best if bundle size is the primary concern or if the team already knows React patterns.

Browser (Preact + Signals)
@preact/signals ↔ Dexie.js
Plotly.js / uPlot charts
ZIP export/import
Python (FastAPI)
Same stateless API surface as A

Pros

  • ~3kb runtime, fastest initial load
  • React ecosystem compatibility
  • @preact/signals is elegant for reactive data

Cons

  • Less opinionated — more architectural decisions on you
  • Fewer data-grid/table components than Vue
  • Smaller community than React or Vue
Common to all three: FastAPI backend is a pure stateless compute engine — no DB, no auth, no file storage. The bodaqs_analysis Python library plugs straight in. All session/artifact data lives in the browser (IndexedDB + localStorage) and is exportable as a ZIP.
