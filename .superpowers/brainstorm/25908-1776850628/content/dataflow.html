Data Flow

From raw CSV upload to rendered charts — every step and where data lives

1
User selects CSV files + picks/configures a preprocess profile
On /preprocess page. Profile config (normalize ranges, butterworth params, schema path) lives in localStorage as a named profile — same structure as the existing suspension_default_v1.json. SHA-256 dedup check runs client-side against the library index to skip already-processed files.
↓
2
POST /api/preprocess — multipart: CSV file + profile JSON
FastAPI endpoint streams the uploaded file into memory, calls run_macro(), returns a structured JSON response. No disk writes on the server.
// Response shape
{
"session_id": "2026-02-20_08-34-26",
"meta": { signals, sample_rate_hz, ... },
"signals": { "time_s": [...], "front_shock_dom_suspension [mm]": [...] },
"events": [ { schema_id, event_type, t_start, t_end, ... } ],
"metrics": [ { schema_id, event_type, metric_name, value, ... } ],
"source_sha256": "abc123..."
}
↓
3
Browser writes artifacts into IndexedDB (Dexie.js)
Three Dexie tables, written in a single transaction:
sessions
session_key (PK)
run_id, session_id
manifest JSON
signals_meta
signals
session_key (PK)
column names
data: Float32Array
(typed arrays, compact)
events / metrics
session_key (index)
schema_id
rows as JSON array
Library index (run list, session list, SHA set) updated in localStorage. Svelte libraryStore reacts automatically.
↓
4
User navigates to session / selects entities for comparison
selectionStore holds the active entity set (mirrors the existing entity_scope widget). Pages derive their data reactively from the store — charts re-render when selection changes. Signal data is lazy-loaded from IndexedDB only when a session window is opened.
↓
5
Export: browser serialises IndexedDB → bodaqs-library-export.zip
ZIP structure mirrors existing artifact folder layout — importable into notebooks. Import does the reverse: unzip → write into Dexie → rebuild library index. A second user can receive the ZIP and import it to view the same sessions.
Key feasibility note: The signal data for 18 sessions at ~50k rows × 6 columns ≈ ~22MB as Float32Arrays in IndexedDB — well within browser limits (typically 50–500MB quota). Events + metrics are tiny (<1MB total). The main performance risk is the initial POST /api/preprocess round-trip for large CSV files — a progress indicator and chunked upload handling will be important.
