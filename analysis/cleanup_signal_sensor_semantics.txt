BODAQS analysis cleanup note: sensor semantics

Intent
------
The current analysis semantics use `sensor` for several related but distinct
ideas:

- the physical logger sensor or channel identity
- a signal role such as fork, shock, or wheel travel
- front/rear disambiguation for suspension and wheel signals
- a canonical column-name stem used by older analysis code

This has worked while the project has focused on fork and shock travel, but it
is starting to blur responsibilities now that bike profiles describe
bike/setup-specific transforms and normalization ranges.

Direction
---------
Introduce `end` as the preferred semantic for front/rear bike location:

- `end: "front"`
- `end: "rear"`

Bike-profile selectors should prefer `end` plus `domain`, `quantity`, and
`unit` for front/rear suspension and wheel signals. For example:

  end=rear, domain=suspension, quantity=disp, unit=mm

The existing `sensor` field should now be treated as logger/source identity
only. It is no longer a semantic selector field for event detection,
segmentation, bike-profile matching, or primary-analysis signal selection. If a
legacy logger uses names such as `fork` or `shock`, handle that with an
appropriate log-metadata generic rather than analysis-time aliasing.

Cleanup work still to consider
------------------------------
- Audit remaining display/internal names that still say "sensor" where the
  value is now an end/context. Several public widget control names are preserved
  internally for API stability, but user-facing labels should say "Ends" or
  "event context".
- Decide whether old packaged snapshots under `analysis/packages/` should be
  regenerated or left as historical artifacts.
- Keep the log-metadata `sensors` block as logger/source calibration metadata;
  do not use it for analysis-side semantic matching.

Completed direction
-------------------
- Event detection now requires semantic `inputs`; legacy sensor expansion has
  been removed from the active detector.
- Segment role resolution binds by semantic event context (`end`, `domain`)
  rather than inferred sensor.
- Primary motion-derived outputs use semantic stems such as
  `rear_wheel_disp`, `rear_wheel_vel`, and `rear_wheel_acc`.
- Secondary motion-derived outputs use compact profile suffixes such as
  `_lp5hz`, or an explicit profile `series_suffix` when provided.
